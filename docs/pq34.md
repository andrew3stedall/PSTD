# PQ34 — SLBLOCK boundary diagnosis

## Evidence reviewed

PQ33 captured the complete 32-byte payload currently classified as a table context:

```text
020001000000000092060000000000007c000000000000007a00000000000000
```

PQ32 and PQ33 also showed that the legacy table parser reports one row and two values only because it reads a zero row width and two zero-width column descriptors.

The captured bytes instead match the Unicode SLBLOCK structure in [MS-PST section 2.2.2.8.3.3.1.2](https://learn.microsoft.com/en-us/openspecs/office_file_formats/ms-pst/5182eb24-4b0b-4816-aa3f-719cc6e6b018): `btype=0x02`, `cLevel=0x00`, `cEnt=1`, four zero padding bytes, and one 24-byte Unicode SLENTRY. The corresponding [SLENTRY definition](https://learn.microsoft.com/en-us/openspecs/office_file_formats/ms-pst/85c4d943-0779-43c5-bd98-61dc9bb5dfd6) consists of 8-byte `nid`, `bidData`, and `bidSub` fields.

## Revised requirement

Diagnose the captured payload as an NDB subnode leaf block before changing extraction semantics:

1. validate the SLBLOCK type, level, entry count, and Unicode padding;
2. prove that the complete payload boundary is exactly the 8-byte header plus `cEnt * 24` bytes;
3. expose the first SLENTRY fields as unsigned little-endian values; and
4. explicitly classify the legacy table parse as spurious when both interpretations appear.

## Implementation

`scripts/pq34_slblock_boundary.py` reads `pq33_summary.json`, writes `pq34_summary.json` and `pq34_summary.md`, and records a deterministic next blocker. CI runs it after the PQ33 capture diagnostic.

This PQ is diagnostic-only. It does not follow block IDs or change extraction output.

## Finding for the next run

PQ35 should decode SLBLOCK/SLENTRY in the Rust subnode path and resolve the entry's `bidData` and optional `bidSub` through the BBT. It should classify the resolved data payload independently, preserve depth and cycle limits, and stop passing SLBLOCK bytes to `TableContext::parse_with_report`. Property materialization remains out of scope until the resolved data block is proven to contain a real HN/TC or row-matrix structure.
