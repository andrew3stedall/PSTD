# PQ35 — SLENTRY target resolution

## Evidence reviewed

PQ34 proved that the 32-byte block previously accepted as a table is a complete Unicode SLBLOCK with one SLENTRY:

- `nid=0x692`
- `bidData=0x7c`
- `bidSub=0x7a`

The previous classifier passed every non-synthetic payload of at least eight bytes to `TableContext::parse_with_report`. A zero row width allowed this SLBLOCK to report one row and two empty values even though it contained no table header or column descriptors.

## Revised requirement

Decode the confirmed SLBLOCK in the Rust subnode path before attempting table parsing:

1. recognize the Unicode SLBLOCK header and entry boundary;
2. parse every complete SLENTRY and enqueue nonzero `bidData` and `bidSub` targets through the existing BBT-backed payload loader;
3. retain the existing depth and visited-block guards;
4. expose SLBLOCK entry and target counts in layout status; and
5. preserve extraction output while measuring the resolved target payload.

## Implementation

`src/pst/subnodes.rs` now classifies Unicode SLBLOCKs before table contexts. SLENTRY data and subnode BIDs become child block references consumed by the existing bounded breadth-first loader. A repeated `bidSub`, including a self-reference, is ignored by the existing visited-block set.

Unit coverage verifies the PQ34 byte shape, target extraction, BBT resolution, and cycle protection. `scripts/pq35_slentry_resolution.py` writes deterministic JSON and Markdown evidence after the public PST fixture run.

## Observed current-fixture result

Corrected CI run 374 passed and reported:

- `pq35_slblock_entries=2`
- `pq35_data_references=2`
- `pq35_sub_references=1`
- `pq35_decoded_blocks=4`
- `pq35_recursive_child_decodes=3`
- `pq35_table_rows_after_resolution=0`
- `pq35_table_values_after_resolution=0`
- `pq35_cycle_limit_preserved=true`

The traversal therefore resolved a second SLBLOCK and both data targets, not only the first entry visible in PQ34's bounded capture. The false one-row/two-value result disappeared.

The first resolved payload currently admitted by the fallback table parser is 902 bytes and begins:

```text
b362ff1caa41414141414141a6a8a841264141411ce8234123413841c8414c41
```

It produces implausible declared counts (`41964` columns and `24128` rows) but zero parsed rows and values. This is evidence that the fallback table admission rule remains too permissive, not evidence of a table.

## Finding for the next run

PQ36 should capture and classify each resolved data payload before assigning table semantics. It should first test for a valid Heap-on-Node header and client signature, record exact payload provenance/BID, and reject implausible flat-table declarations before distinguishing a TC header, row matrix, property context, encoded payload, or unsupported payload. Property materialization remains out of scope until a resolved payload boundary is proven.
