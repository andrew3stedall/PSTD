# Controlled ANSI PST fixture generation

## Purpose

Vertical 40 requires a real Outlook 97-2002 PST before ANSI BBT/NBT traversal can be enabled. Synthetic headers prove field layout only. This procedure creates a controlled, non-private ANSI PST whose bytes can be pinned and measured without relying on unlicensed mailbox data.

The generated file is **fixture evidence**, not proof of broad ANSI compatibility. Until its exact bytes pass a dedicated fail-closed workflow, ANSI traversal and extraction remain unsupported.

## Requirements

- Windows with classic desktop Microsoft Outlook installed.
- An Outlook profile that can start without interactive first-run prompts.
- PowerShell 5.1 or later.
- A clean working directory outside any private mailbox location.

The generator uses Outlook `NameSpace.AddStoreEx` with `olStoreANSI` (`3`). Microsoft documents this value as creating the legacy ANSI personal-folders format compatible with Outlook 97-2002.

## Generate

Run from the repository root in an isolated Windows environment:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass `
  -File .\scripts\generate-ansi-fixture.ps1 `
  -OutputPath .\artifacts\pstd-ansi-baseline.pst
```

The script refuses to overwrite an existing file. It creates only controlled synthetic content:

- folder `/Synthetic Mail`;
- nested folder `/Synthetic Mail/Nested`;
- one plain-text message with To, Cc and Bcc recipients;
- one message with independent plain and HTML bodies plus a by-value text attachment;
- folder `/Typed Non-Mail` containing one contact that must never be coerced into EML.

After detaching the store, the script verifies:

- file magic `!BDN`;
- PST client signature `SM`;
- NDB version `14` or `15`;
- exact byte length and SHA-256.

It prints those values as JSON. Do not commit the PST until the output is reviewed and the fixture licence is recorded.

## Admission record

Before committing generated bytes, add a manifest containing at least:

```yaml
name: pstd-ansi-baseline.pst
format: Microsoft Outlook PST 97-2002 ANSI
generator:
  script: scripts/generate-ansi-fixture.ps1
  repository_commit: <full commit SHA>
  windows_version: <exact Windows version>
  outlook_product: <exact Outlook product and build>
  powershell_version: <exact PowerShell version>
content: controlled synthetic data only
licence: CC0-1.0
byte_length: <exact integer>
sha256: <64 lowercase hexadecimal characters>
header:
  magic: "!BDN"
  client_signature: "SM"
  ndb_version: <14 or 15>
```

The generated fixture should be accompanied by an explicit CC0 dedication from the person running the generator. Repository code licensing alone is not sufficient evidence for generated mailbox bytes.

## First fixture-level acceptance result

The first workflow must verify the pinned hash before invoking PSTD. With ANSI traversal still disabled, the exact expected result is:

- ANSI header detected;
- version-specific 32-bit NBT and BBT roots reported diagnostically;
- extraction classified as unsupported;
- zero folders, messages, recipients, bodies, attachments and EML emitted;
- no Unicode fixture metric changes.

Only after this baseline passes should a separate milestone admit one bounded ANSI root page. Do not reuse Unicode page widths, entry widths, trailer offsets or BID/IB decoding.

## Fail-closed boundaries

Reject the fixture or generation run when:

- Outlook creates Unicode version 23 instead of ANSI version 14/15;
- Outlook was connected to or populated from a private mailbox;
- provenance, product build, byte length, hash or licence is missing;
- the file changes after hashing;
- PSTD emits objects before ANSI traversal is explicitly implemented and reviewed.
