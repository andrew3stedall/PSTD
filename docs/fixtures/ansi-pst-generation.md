# Controlled ANSI PST fixture generation

## Purpose

Vertical 40 requires a deterministic version-14 or version-15 PST before ANSI BBT/NBT traversal can be enabled. The fixture must contain only controlled synthetic data, have an explicit redistribution basis, and be reproducible on Linux.

The selected approach is a **fixture-only Rust emitter**, not a general PST writer. It exists solely to produce one small, immutable ANSI compatibility fixture whose exact bytes can be independently validated and pinned. It must not become part of PSTD's public extraction API.

## Portability boundary

The complete generation and validation path must run on Linux using Rust and normal repository tooling. Windows, Outlook, COM, PowerShell, proprietary SDKs, private mailbox data, and manual binary editing are not required.

The generated bytes are fixture evidence, not proof of broad ANSI compatibility. ANSI extraction remains disabled until each admitted structure is validated against the format specification and an independent reader.

## Why a fixture-only emitter is acceptable

Obtaining an externally created ANSI PST has produced repeated licensing and provenance blockers. A narrowly scoped emitter avoids those blockers because:

- all mailbox content is defined in the repository;
- the generated bytes can be dedicated under the repository's fixture licence;
- generation is deterministic and reviewable;
- malformed or unsupported layouts can be rejected before admission;
- the implementation can stop after the single required fixture shape rather than growing into a general writer.

The emitter must not share parsing code with the reader for validation-critical fields. Shared constants are acceptable only when independently asserted against published format offsets and widths.

## Staged implementation

### Stage A: structural ANSI baseline

Create a small Rust binary under `tools/` that emits a deterministic version-14 PST containing:

- a valid ANSI header;
- one empty NBT leaf root page;
- one empty BBT leaf root page;
- valid page trailers, signatures and CRC values;
- no allocated message blocks and no extracted objects.

Acceptance requires:

- identical SHA-256 across repeated Linux runs;
- exact byte length and header fields;
- PSTD classifies the file as ANSI and remains fail-closed;
- an independent open-source reader accepts the container structure or reports only the expected empty-store condition;
- zero folders, messages, bodies, recipients, attachments, typed non-mail objects and EML.

Stage A unlocks bounded ANSI root-page decoding. It does not claim email compatibility.

### Stage B: one-folder, one-message fixture

Extend the fixture-only emitter with the minimum structures required for:

- one root message store;
- one folder named `Synthetic Mail`;
- one `IPM.Note` message;
- deterministic subject, sender, To recipient and plain-text body;
- no attachment and no embedded message.

The exact synthetic values must use reserved example domains and fixed timestamps. The first EML acceptance result must lock headers, CRLF body bytes, output path, byte length and SHA-256.

### Stage C: compatibility breadth

Only after Stage B passes, add separate controlled variants for:

- independent HTML and plain bodies;
- one by-value attachment;
- To/Cc/Bcc;
- one typed contact that is explicitly classified as non-mail;
- malformed page, block and ownership variants that must fail closed.

Each variant should be a separate fixture or explicit generator mode so one failure cannot be hidden by unrelated content.

## Determinism requirements

The emitter must pin every field that can vary between runs:

- NDB and client versions;
- unique values and allocation identifiers;
- NIDs, BIDs and IB offsets;
- timestamps;
- code page;
- page and block ordering;
- padding bytes;
- CRC and signature inputs.

Generation must refuse to overwrite an existing fixture unless an explicit output flag is supplied. CI must regenerate into a temporary directory and compare byte-for-byte with the committed fixture.

## Independent validation

A fixture generated by PSTD-owned code cannot validate itself. Admission therefore requires three independent layers:

1. byte-level checks implemented in a small test module that does not call the production parser;
2. PSTD's normal diagnostic and extraction path;
3. a pinned Linux open-source reader, initially `libpff`/`pffinfo` or another redistributable implementation, used only as a fixture validator.

Disagreement between the emitter, byte-level validator, PSTD or the independent reader blocks fixture admission. No implementation should be relaxed merely to make the generated fixture pass.

## Admission record

Each committed fixture must include a manifest containing:

```yaml
name: pstd-ansi-stage-a.pst
format: Microsoft Outlook PST ANSI
ndb_version: 14
generator:
  crate_or_binary: <path>
  repository_commit: <full commit SHA>
content: controlled synthetic data only
licence: CC0-1.0
byte_length: <exact integer>
sha256: <64 lowercase hexadecimal characters>
header:
  magic: "!BDN"
  client_signature: "SM"
  crypt_method: 0
validation:
  pstd: <exact result>
  independent_reader: <tool and pinned version>
```

## Fail-closed boundaries

Reject generation or admission when:

- any offset, count, identifier, CRC or signature is inferred rather than explicitly constructed;
- two structures claim the same NID, BID, IB range or ownership relationship;
- a page or block exceeds its declared range;
- the independent reader disagrees with the declared container structure;
- repeated generation changes any byte;
- PSTD emits objects not explicitly present in the fixture manifest;
- Stage A data is treated as evidence of message or EML compatibility.

## Out of scope

This work does not create a public PST-writing API, support arbitrary user-supplied content, import EML into PST, reproduce Outlook behaviour, or attempt full ANSI allocation-map maintenance. The emitter should remain small, deterministic and fixture-specific.
