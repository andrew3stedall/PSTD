# PQ33 — table payload prefix capture

## Evidence reviewed

PQ32 confirmed that the public PST fixture still parses one table row and two values while every descriptor offset and width field remains zero. That means the current eight-byte table descriptor assumption is not structurally supported.

## Revised requirement

Capture the selected table-like subnode payload before interpreting it. The capture must be deterministic, bounded, and diagnostic-only:

1. expose the selected table-like payload byte length;
2. expose a capped hexadecimal prefix from the same payload;
3. expose whether the prefix was truncated; and
4. validate the artifact shape without changing extraction output.

## Implementation

`src/pst/subnodes.rs` now records `subnode_table_payload_byte_len`, `subnode_table_payload_prefix_byte_len`, `subnode_table_payload_prefix_truncated`, and `subnode_table_payload_prefix_hex` for the first table-like subnode layout. The prefix is capped at 32 bytes.

CI also extracts `data/messages.jsonl` from the manifest tar into the progress artifact as `pq33_messages.jsonl`, because the table-like payload prefix is emitted at message/subnode layout level. `scripts/pq33_payload_prefix.py` reads `run_summary.json` plus that message-status file, writes `pq33_summary.json` and `pq33_summary.md`, and verifies that the captured prefix has the expected hex length.

## Expected current-fixture result

The sample fixture should classify as:

- `pq33_payload_prefix_visible=true`
- `pq33_prefix_shape_valid=true`
- `pq33_next_blocker=table_payload_header_boundary_interpretation`

## Finding for the next run

PQ34 should interpret the captured prefix as raw bytes and identify plausible heap/table-context header boundaries before changing table parsing semantics. The next step should compare the prefix against known PST heap-on-node and BTH/table-context signatures and should still avoid property materialization until the payload boundary is proven.
