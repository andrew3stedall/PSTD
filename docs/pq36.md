# PQ36 — decoded payload admission

## Evidence reviewed

PQ35 resolved two SLENTRY data targets, but the first 902-byte payload began with `b362ff1c...` and was accepted as a flat table with impossible declarations of 41,964 columns and 24,128 rows.

The fixture's Unicode PST header contains `bCryptMethod=0x01`. Per [MS-PST HEADER](https://learn.microsoft.com/en-us/openspecs/office_file_formats/ms-pst/c9876f5a-664b-46a3-9887-ba63f113abf5), this is `NDB_CRYPT_PERMUTE`, and non-internal data blocks must be decoded with the [permutative algorithm](https://learn.microsoft.com/en-us/openspecs/office_file_formats/ms-pst/5faf4800-645d-49d1-9457-2ac40eb467bd).

Applying the inverse permutation to the captured prefix produces:

```text
5203ec7c4000000000000000b5040400600000007c0e30003000320034002000
```

This exposes the expected Heap-on-Node signature `0xec`, table-context client signature `0x7c`, and user root `0x40`.

## Revised requirement

1. expose `bCryptMethod` in header diagnostics;
2. decode external blocks for `NDB_CRYPT_PERMUTE` while leaving internal blocks unchanged;
3. reject unsupported crypt methods safely;
4. classify valid Heap-on-Node payloads by client signature before table parsing;
5. admit the legacy flat-table format only when descriptor and row bounds are structurally valid; and
6. preserve BID, length, and prefix evidence for unsupported resolved payloads.

## Implementation

`src/pst/payload.rs` now decodes non-internal permutatively encoded blocks using the inverse MS-PST permutation. `src/pst/subnodes.rs` recognizes valid HN table, property, and BTH client signatures before applying a strict bounded flat-table fallback. Unit tests cover external decoding, unchanged internal blocks, HN classification, and rejection of implausible declarations.

`scripts/pq36_payload_admission.py` records crypt method, heap classifications, rejected payload provenance, and the next blocker in the public fixture artifact.

## Observed current-fixture result

Corrected CI run 378 passed and confirmed:

- `pq36_crypt_method=1`
- `pq36_heap_contexts=1`
- `pq36_heap_table_contexts=1`
- `pq36_permute_decode_visible=true`
- `pq36_implausible_table_rejected=true`
- `pq36_table_rows=0`
- `pq36_table_values=0`

The decoding correction also materially improved message extraction:

- selected properties increased from 0 to 16;
- unknown properties fell from 74 to 19;
- one text-body property and one RTF-body property were recovered;
- body payload records increased from 0 to 2; and
- fallback body records fell from 1 to 0.

One resolved payload remains structurally unsupported:

- BID: `116` (`0x74`)
- length: `208` bytes
- prefix: `2d0000000b000000c0000000e000000080000000a00000000100000000010000`

This payload is no longer misreported as a table. Its shape should be interpreted only after the decoded HN table header identifies whether it is the row matrix or another referenced structure.

## Finding for the next run

PQ37 should traverse the decoded Heap-on-Node table context: resolve `hidUserRoot`, parse TCINFO and column descriptors, then identify the RowIndex and row-data HNID/subnode boundary. It should correlate the unresolved BID `0x74` payload with the TC row-data reference without assuming that relationship in advance. It must retain bounds checks and remain diagnostic-only until real rows and property tags are verified.
